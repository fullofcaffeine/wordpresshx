# Provisional generated-output guidance

This document explains the proposed ADR-020 model. It is not legal advice, a
license grant, or permission to publish an artifact.

## The practical rule

Generated output is classified by where its bytes and expressive content came
from, not merely by the executable that wrote the file. WordPressHx therefore
must not stamp one inferred license onto every generated PHP, TS, JS, CSS, JSON,
or markup file.

For each final artifact, the build manifest must separate at least:

1. **User-authored or user-configured content.** Haxe declarations, inline HXX,
   text, assets, raw escape-hatch code, and configuration supplied by the
   project. WordPressHx does not claim ownership of this input.
2. **Compiler-emitted original boilerplate.** Small structural output created by
   the WordPressHx emitter. Its final treatment is part of the pending
   repository license decision.
3. **Toolchain runtime or standard-library portions.** Any substantial Haxe,
   Genes, PHP-target, HXX, or other helper implementation copied into output.
   Each portion must retain its origin, exact version, content hash, license, and
   required notice.
4. **Third-party or upstream-derived portions.** WordPress/Gutenberg contracts,
   external templates, vendored libraries, copied examples, and other material
   not authored solely by the project owner.

Combining those categories into a plugin, theme, or site can create additional
compatibility and distribution obligations. The package owner must confirm the
rights and final terms for the complete artifact.

## HXX and `return <markup>`

Inline HXX is an authoring syntax. A normal typed component or template should
eventually be able to write `return <main>...</main>`; the compiler lowers that
markup at compile time into native PHP/HTML for server output or Genes TSX/JS for
browser output. No HXX parser, component registry, VDOM, or template runtime is
intended to ship merely because HXX was used.

That ergonomic lowering does not erase origin. Literal text and expressions
remain attributable to project input; generated structure is emitter output;
any copied escaping helper, runtime support, standard-library implementation, or
template remains separately attributable. Escape hatches must be classified the
same way and cannot silently override license metadata.

## Required artifact manifest

Before publication, each packed Haxelib/npm/source/WordPress artifact must bind:

- the exact source commit and deterministic artifact digest;
- every included file and its origin category;
- content hashes for copied boilerplate, runtime, standard-library, template,
  scaffold, and third-party portions;
- exact component/version/source identifiers and SPDX expression or unresolved
  status;
- applicable copyright notices and full license texts;
- generated-versus-copied boundaries, including code eliminated at compile time;
- an SBOM and provenance receipt tied to the final bytes;
- product-owner and qualified-review approval for that exact notice set.

If a generator cannot classify a portion, it must emit an unresolved origin and
the publication gate must fail. A project-level raw override may not convert
unknown evidence into an accepted license conclusion.

## Current boundaries

Current compiler/HXX fixtures are evidence-only. They do not establish the
contents or licensing of a future public generated artifact. In particular:

- `reflaxe.php` is internal version `0.0.0` and has imported GPL-2.0-or-later
  provenance from `wordpresshx-port`;
- Genes is an exact build input, not yet a shipped WordPressHx dependency;
- the HXX parser closure is compile-time-only and prohibited from generated
  runtime output by ADR-011;
- Haxe standard-library/runtime copying has not yet been inventoried per target;
- no scaffold, installable plugin/theme/site ZIP, or public package currently
  exists to audit.

The official Haxe Foundation statement distinguishes its compiler and standard
library licenses, and the GNU GPL FAQ explains why compiler output generally
depends on input while copied program text can matter. Qualified review must
apply those principles to actual generated bytes rather than this abstraction.
