# Todo data-store lab

This runnable example adds a typed application spine to the native Gutenberg
sidebar. Three small tasks, review state, priority, and an asynchronous sync
rehearsal live in an immutable Haxe state model. WordPress' real
`@wordpress/data` registry owns registration, dispatch, selection,
subscription, and React updates.

The useful difference from plain JavaScript is visible before the browser
runs: the store key, state, reducer, action discriminator, dispatched actions,
selectors, and hooks form one compiler-checked contract. If they disagree, the
Haxe compiler reports the source location. There is no stringly typed action
bag and no application-authored `any` escape hatch.

This is the SDK-064 foundation for Todo Studio, not yet the complete flagship.
It deliberately uses in-memory editor state. WordPress persistence, REST
authorization, packaging through the project CLI, and the full site-facing
Todo experience remain later integrated milestones.

## Run the complete proof

Prerequisites are Docker with Compose v2, Haxe 4.3.7 through Lix 15.12.4, Node
22.17.0 with npm 10.9.2, Python 3, and Git. From the repository root:

~~~sh
SDK064_VISUAL_OUTPUT=packages/gutenberg/sdk064-preview \
  bash packages/gutenberg/scripts/test-data-store.sh
~~~

Expect the final line:

~~~text
SDK-064 typed WordPress data-store gate passed
~~~

The gate proves all of these layers:

1. the exact WordPress 7.0 `@wordpress/data` source and npm package match the
   immutable profile;
2. the Haxe source contains no `Dynamic`, `Any`, `cast`, `Reflect`, or
   `untyped`;
3. reducer, action, selector, immutability, loading, error, and recovery tests
   pass twice;
4. invalid store definitions fail at compile time for the expected reason;
5. two Genes compilations and two official WordPress builds have identical
   generated bytes;
6. emitted TSX passes strict TypeScript without owned `any` or `unknown`;
7. the generated native plugin passes PHP 7.4 and 8.4 syntax checks; and
8. exact WordPress 7.0 plus Chromium proves keyboard and mouse actions, native
   subscription updates, async failure/retry, post-type gating, focus behavior,
   public unregister, accessibility, and zero console/page errors.

The temporary site is removed automatically. The retained
`packages/gutenberg/sdk064-preview/` directory contains the generated plugin,
semantic asset plan, screenshot, and browser evidence JSON. Use the shorter
build-only lane while iterating on pure code:

~~~sh
bash packages/gutenberg/scripts/test-data-store.sh --skip-wordpress
~~~

To leave an interactive local site running after the check:

~~~sh
bash packages/gutenberg/scripts/start-example-server.sh todo-data-store-lab
~~~

The script prints the exact post-editor URL and local test credentials. Open
the editor Options menu, choose **Todo data-store lab**, and exercise the task,
review, priority, and failure/retry controls. Stop and remove only this
example's isolated database with:

~~~sh
bash packages/gutenberg/scripts/start-example-server.sh todo-data-store-lab stop
~~~

## Follow the source from model to UI

- [`TodoDomain.hx`](../../packages/gutenberg/test/data-store-fixture/src/sdk064/fixture/TodoDomain.hx)
  defines the immutable state, closed action variants, constructors, pure
  reducer, and selector.
- [`TodoStore.hx`](../../packages/gutenberg/test/data-store-fixture/src/sdk064/fixture/TodoStore.hx)
  defines and registers the validated store, then gives the app domain-specific
  command names.
- [`Main.hx`](../../packages/gutenberg/test/data-store-fixture/src/sdk064/fixture/Main.hx)
  reads the native reactive snapshot and returns the editor UI as inline HXX.
- [`DataStore.hx`](../../packages/gutenberg/src/wordpress/hx/gutenberg/data/DataStore.hx)
  is the reusable public compile-time entry point.

The normal definition is intentionally small:

~~~haxe
private static final key =
  StoreKey.literal("wordpresshx/todo-studio-lab");

private static final store:TypedDataStore<TodoState, TodoAction> =
  DataStore.define(key, TodoDomain.initial(), TodoDomain.reduce);
~~~

`DataStore.define` checks that the reducer accepts the declared state and
closed action shape, returns the same state type, and has a string-compatible
`type` discriminator. `StoreKey.literal` checks the namespaced key. Normal
operations remain typed descriptors:

~~~haxe
DataStores.register(store);
final current:TodoState = DataStores.snapshot(store);
DataStores.send(store, TodoDomain.toggleTask("invite-review"));
~~~

Passing a string or another store's action to that final call does not produce
a late Redux error; Haxe rejects the program.

## Try a safe change

Add a new action variant to `TodoActionType`, an action constructor, and a
matching reducer branch. Haxe's closed abstract and the typed store keep the
change connected. Then add a pure unit assertion and an HXX control. If you
forget a type or send the wrong action, the compiler stops at the inconsistency
before the native WordPress build begins.
