# PHP source correlation

WordPressHx emits readable native PHP and preserves its native exception text.
SDK-025 adds an authenticated diagnostic layer that can correlate selected PHP
frames back to exact Haxe ranges without changing WordPress exception behavior.
It is a WordPressHx range-map contract, not a claim that PHP implements browser
Source Map v3.

## Artifact flow

```text
Haxe source + semantic IDs
          |
          v
generic reflaxe.php printer -----> readable generated PHP
          |                              |
          +---- exact byte spans --------+
                         |
                         v
             *.php.haxe-map.json
                         |
WordPress package inventory ------+
                         |
                         v
                 source-index.json
                         |
native Throwable text --> wphx-sdk trace php --> native + correlated frames
```

The generic compiler owns exact source identity, half-open UTF-8 byte spans,
semantic mapping records, trace-anchor intent, deterministic rendering, and a
caller-named neutral map writer. It contains no WordPress paths or semantics.
The WordPress build layer supplies the public
`wordpresshx.php-haxe-range-map.v1` identity and writes the exact package source
index. The Haxe/Genes CLI validates and consumes both.

## Why line anchors exist

A normal PHP stack frame supplies a file and line but no expression column. A
generated line can contain more than one semantic range, so the CLI does not use
nearest-line or smallest-range guessing. The emitter explicitly designates at
most one statement mapping as the runtime trace anchor for a line. Exact path,
generated-content hash, map integrity, and unique anchor must all agree before a
frame is labeled `mapped-trace-anchor`.

The original native frame remains in both text and JSON output. Frames without a
valid anchor remain visible with an unmapped status. Private helper failures can
therefore produce one exact throw frame and additional honest native-only
frames.

## Profiles and retention

The same generated PHP bytes are exercised in three profiles:

- `development`: local index/map and an allowlisted source copy;
- `packaged-evidence`: production PHP plus a separate map/index companion, with
  source content omitted; and
- `production-plugin`: the five ordinary installable PHP files only.

The deterministic production ZIP contains no map, index, Haxe source, inline
map, or development handler. Its opt-in debug companion contains only the map
and source index and is bound to the production adapter PHP hash. Shipping any
source content later remains subject to explicit allowlisting, secret scanning,
path scanning, and license review.

## Evidence boundary

The fixture deliberately fails through an action hook, a REST callback, a
dynamic-block render callback, and a public entry into a private helper. It runs:

- directly on exact PHP 7.4.33 and 8.4.7 images with networking disabled;
- after real plugin activation on clean WordPress 7.0 with both MySQL and
  MariaDB; and
- through eight development/packaged CLI traces in locked Node 22.17.0.

Tampered map/generated bytes, dishonest coordinates, unknown fields, unsafe
paths, and ambiguous anchors fail closed. Basename-only input stays unmapped.
This is bounded source-correlation and packaging evidence; it is not general
production support, arbitrary-Haxe PHP compilation, a production exception
handler, browser correlation, or package-publication authorization.
