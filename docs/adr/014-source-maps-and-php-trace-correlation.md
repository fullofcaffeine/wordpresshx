# ADR-014: Source maps and PHP trace correlation

- Status: accepted
- Date: 2026-07-18
- Owners/reviewers: Marcelo Serpa (product and architecture owner); Codex (compiler-family, integrity, privacy, and evidence review)
- Bead: `wordpresshx-adr-014`
- Profiles/layers: generic PHP compiler; SDK source index and trace CLI; Genes TS/TSX; WordPress browser builds; development/debug/production packaging
- Supersedes: none
- Superseded by: none

## Context

WordPressHx needs failures in generated PHP and browser bundles to lead a Haxe
developer back to useful source without hiding the native WordPress, PHP, or
browser evidence. That is a source-correlation problem, not a promise that every
target runtime has the same stack model or that one source-map format is suitable
for every target.

The two target families start from different facts:

- Genes emits standard Source Map v3 mappings from generated TypeScript/TSX or
  classic JavaScript to Haxe. A bundler emits another standard map from final
  JavaScript to generated source. Standards-based composition is possible, but
  it is only trustworthy for an exact compiler, bundler, configuration, entry,
  and mode after a runtime throw proves it.
- PHP stack frames normally provide a generated filename and line, not a byte
  range or column. PHP has no universally consumed browser-style source-map
  protocol. Readable native PHP therefore remains essential even when private
  correlation data exists.

The repository has useful but bounded evidence. SDK-021 records validated
repository-relative Haxe declaration ranges and exact generated PHP declaration
line ranges; it explicitly does not serialize a map or implement trace lookup.
SDK-032 proves that two bounded esbuild bundles and the generated TSX map back to
Haxe tokens with zero machine-path leaks. It does not deliberately throw through
both development and minified production stacks, and it is not proof for every
WordPress build adapter. SDK-033 proves the exact official
`@wordpress/scripts` asset lane, but deliberately leaves end-to-end source
correlation to SDK-034. In particular, that lane removed the Genes first-layer
map because the official loader could otherwise seek a host Haxe path. None of
these receipts supports a general “full source maps work” statement.

The full-port reference contains useful bounded examples of normalized native
PHP traces, stock-Haxe source comments, and original-path shell segment records.
It does not provide the content-bound, path-safe, versioned lookup contract this
SDK needs, and its runtime topology is not imported.

The source-map lookup contract is a public API under the PRD. Its coordinates,
path identities, selection rules, integrity failures, retention policy, and
machine-readable trace results therefore need an accepted decision before
SDK-025 and SDK-034 implement broader behavior.

## Decision

### One source index, target-appropriate map layers

The SDK uses two versioned JSON contracts:

- [`php-haxe-map.schema.json`](../../schemas/php-haxe-map.schema.json) defines
  `wordpresshx.php-haxe-range-map.v1`, emitted next to an authenticated
  generated PHP file as `*.haxe-map.json`.
- [`source-correlation-index.schema.json`](../../schemas/source-correlation-index.schema.json)
  defines `wordpresshx.source-correlation-index.v1`, one package-build index
  over PHP maps, browser map layers, logical source roots, exact file hashes,
  retention, and proof receipts.

Browser layers remain Source Map v3. The SDK does not invent a proprietary
browser-map encoding. The package index content-binds those standard maps to
the exact generated, intermediate, and source files needed for sequential or
composed lookup.

The accepted machine-readable decision is
[`source-correlation-architecture.json`](../../manifests/source-correlation-architecture.json).
Schema version 1 and the lookup rules in this ADR are public contracts. A
breaking interpretation change requires a new format version and an explicit
migration; changing a JSON Schema in place cannot silently reinterpret existing
maps.

### Coordinates and content identity

All PHP ranges use this coordinate system:

- UTF-8 bytes are authoritative;
- byte ranges are half-open `[startByte, endByte)` and non-empty;
- lines are one-based;
- columns are zero-based UTF-8 byte columns; and
- line/column endpoints are redundant diagnostics that must be recomputed and
  validated against the exact bytes before lookup.

Every generated PHP record contains its normalized relative path, SHA-256,
UTF-8 byte length, line count, encoding, and LF line-ending policy. Every Haxe
or admitted native source record also contains a SHA-256, byte length, and line
count. A formatter, postprocessor, manual edit, or packaging change that alters
the generated bytes invalidates the old map. Lookup never continues using stale
line numbers.

This deliberately differs from the current SDK-021 `PhpSourceRange`, whose
one-based declaration columns and generated inclusive line ranges are internal
foundation evidence. SDK-025 must derive the v1 byte spans from authenticated
source and printer bytes; it must not relabel the old fields as v1 coordinates.

### Logical paths only

Maps and indexes store normalized POSIX logical paths under declared root IDs,
not filesystem roots. They reject:

- absolute and drive paths;
- backslashes, control characters, empty segments, `.` and `..` segments in
  stored logical identities;
- usernames, temporary roots, and host checkout paths;
- basename-only file selection; and
- nearest-file or suffix guessing.

Standard Source Map v3 `sources` entries may use relative URL traversal from a
map to a file, because that is normal map syntax. Before use, the SDK resolves
each entry within the indexed artifact namespace and requires it to equal the
complete indexed file identity and content hash. Resolution outside that
namespace fails.

Local source trees are supplied at lookup time as explicit root mappings such
as `project=/work/site`. They are never serialized into reproducible artifacts.
An unresolved root can still yield a logical Haxe location, but the CLI does not
invent a local path or open a different file with the same name.

### PHP range-map format

A PHP map is emitted in the same artifact transaction as the exact PHP bytes it
describes. It contains deterministic, sorted mappings with stable mapping IDs,
generated spans, node kinds, structural depth, and one of three origins:

- `haxe-source`: exact authenticated Haxe span plus a stable semantic node ID;
- `native-source`: an admitted native template or adapter source span plus a
  stable semantic node ID; or
- `compiler-generated`: no fabricated Haxe position, but a closed reason class,
  stable reason ID, and optional parent semantic identity.

The generic compiler owns byte provenance and neutral semantic identities.
WordPress semantic-plan IDs are passed into that neutral contract by the SDK
build/profile layer; `reflaxe.php` does not import WordPress concepts. Crossing
overlaps are invalid. Properly nested mappings are allowed for declarations,
members, statements, expressions, adapters, and markup.

For a caller that has an exact generated byte or column, lookup:

1. resolves the complete logical generated filename;
2. authenticates its bytes against the map and package index;
3. validates every stored endpoint against those bytes;
4. selects mappings containing the requested byte;
5. chooses the smallest containing byte range;
6. then chooses the greatest structural depth; and
7. rejects a remaining tie as an invalid or ambiguous map.

It never chooses a nearby range.

PHP native stacks usually have only a line. Selecting the smallest expression on
that line would imply precision the runtime did not provide. The emitter must
therefore create at most one `emitter-runtime-line` trace anchor for each line it
can correlate honestly to an executable semantic mapping. Line-only lookup uses
that exact anchor. A missing anchor returns `unmapped-no-anchor`; a duplicate or
invalid anchor rejects the map. This makes partial coverage visible and keeps a
formatting line, multi-expression line, or helper line from being guessed back
to unrelated Haxe.

Fine-grained source comments are optional development diagnostics only when they
cannot disturb a plugin header, direct output, mixed PHP/HTML, or native behavior.
They are never lookup or ownership authority. A production public-file generator
header may contain logical module/generator identities, but default production
output omits fine-grained comments and all host paths.

### Package source index

The SDK build layer emits a package source index for one exact build. It records:

- SDK, package, exact profile, and build-input identities;
- a sorted content-addressed inventory of runtime files, generated source, maps,
  and optionally distributed source;
- logical source roots and how an offline caller may resolve them;
- retention policy for the index, maps, source content, and development handler;
- one correlation record per entry artifact;
- ordered map layers with exact generated and source file IDs; and
- proof receipt IDs and the evidence level for that exact strategy.

The admitted strategies are:

- `php-range-map`: one PHP v1 map layer;
- `browser-composed-v3`: one proven final-JavaScript-to-Haxe Source Map v3 layer;
- `browser-two-stage-v3`: final JavaScript to generated TS/TSX followed by the
  Genes TS/TSX-to-Haxe map; and
- `unavailable`: an explicit reason instead of a fake or stale map.

For a two-stage browser record, the source file identity from one layer must be
the exact generated file identity of the next layer. Array order, matching
basenames, or source-map comments alone do not establish continuity. An index
may contain a mixture of composed, two-stage, and unavailable entries because
admission is per exact entry and build mode.

### Browser composition and fallback

Genes owns Haxe-to-generated-source semantics and continues to emit standard
Source Map v3. The selected WordPress build adapter owns final JavaScript-to-
generated-source maps. WordPress package names, entry identities, modes, and
debug-package retention remain SDK concerns; no WordPress branch is added to
Genes.

The build may label an entry `browser-composed-v3` only when the exact profile,
Genes version, build adapter version/configuration, entry, and mode pass all of
these checks:

- each input map validates independently;
- standards-based composition produces deterministic bytes;
- the index binds the map and every layer to exact content hashes;
- a deliberate development throw at a named Haxe token resolves back to that
  token;
- a deliberate minified production throw resolves to the same expected Haxe
  token;
- no absolute/machine path leaks; and
- no unmapped or partially composed segment is silently presented as complete.

Successful composition for one esbuild fixture is not blanket proof for the
official `wp-scripts` lane or a future adapter. If any admission check fails, the
entry uses `browser-two-stage-v3`: retain and authenticate both layers, then let
the CLI resolve JavaScript to generated TS/TSX and TS/TSX to Haxe sequentially.
If either individual layer is invalid or unavailable, record `unavailable`.

The default is an honest two-stage workflow, not an assumption that every tool
will recursively compose maps. A future Genes source-map defect is handled in an
isolated Genes worktree as a generalized compiler fixture and full upstream
regression before a Genes PR/release and a new immutable SDK pin; it is not fixed
with WordPress-specific compiler logic here.

### Trace CLI contract

The implementation commands are:

```text
wphx-sdk trace php <stack-file> --index <source-index> [--source-root <id>=<path>] [--format text|json]
wphx-sdk trace browser <stack-file> --index <source-index> [--source-root <id>=<path>] [--format text|json]
```

The trace library is offline, read-only, and performs no network lookup. Before
mapping a frame it validates the index schema, exact map and generated file
hashes, map coordinates, file identities, and chain continuity. Text output
shows the untouched native frame followed by correlation. JSON output is stable
and contains both native and correlated fields.

Per-frame results use a closed vocabulary:

- `mapped-exact-range`;
- `mapped-trace-anchor`;
- `mapped-composed`;
- `mapped-two-stage`;
- `native-unmapped`;
- `unmapped-no-anchor`; or
- `unmapped-no-layer`.

Unmapped native frames are valid results and do not make an otherwise valid
trace invocation fail. Invalid input/usage exits 2, failed integrity or schema
validation exits 3, and an ambiguous contract exits 4. A processed trace exits
0 even when it contains honest unmapped frames. The CLI never rewrites the
original stack file, suppresses native frames, or reports a lower-confidence
result as exact.

SDK-025 owns PHP stack parsers, human/JSON snapshots, public/private deliberate
failure fixtures, and the concrete trace library. SDK-034 owns browser stack
parsers, standards-based composition, deliberate development/minified throws,
and the two-stage browser path. This ADR fixes their shared contract; its schema
fixture is not those runtime proofs.

### Development, debug companion, and production retention

The default production plugin/theme/site installation artifact contains:

- readable native PHP and normal production browser bundles;
- no PHP maps, browser maps, package source index, Haxe source content, inline
  maps, or development exception handler; and
- unchanged public/native exception and logging behavior.

An opt-in debug companion may be produced separately. It is bound to exact
production artifact hashes and contains external maps plus the source index.
Source content is omitted by default; developers resolve project/dependency
roots locally. Shipping allowlisted source content requires all of:

- an explicit file allowlist;
- Gitleaks/secret scanning;
- machine-path and username scanning;
- license/notices review; and
- the applicable ADR-019 unsafe/security review outcome.

Maps are never inline in production JavaScript or PHP. An explicitly configured
external production browser-map publication is treated as a debug-companion
distribution with the same exact hash, privacy, secret, and license rules; it
does not place maps in the default WordPress install ZIP.

Development builds may emit maps and an index. `sourcesContent` and packaged
Haxe sources remain opt-in rather than a convenience default. Development and
production still use the same typed/profile/ownership semantics; map retention
cannot create a second untyped build path.

### Optional WordPress development handler

An optional WordPress integration may call the same trace library and append a
correlation block to development logs. It is disabled by default, excluded from
production, and must not:

- suppress or replace native PHP frames;
- catch an exception solely to change its response;
- replace WordPress recovery behavior or another registered handler;
- change HTTP status/body behavior; or
- fetch maps or source over the network.

The offline CLI remains the supportable baseline. A runtime handler is a
development convenience, not a requirement for source correlation.

## Rationale

Exact byte/content binding prevents a stale map from producing a persuasive but
wrong Haxe location after formatting or postprocessing. Half-open UTF-8 byte
ranges are unambiguous across multibyte source and can be checked independently
of editor conventions. Redundant line/column data keeps artifacts readable while
remaining verifiable.

The explicit PHP trace-anchor rule reflects the information a native PHP frame
actually carries. It provides less apparent coverage than nearest-line lookup,
but every mapped line has emitter-owned intent. This is more useful operationally
than a high coverage percentage built from guesses.

Standard Source Map v3 remains the right browser interchange. The package index
adds the exact build identity and layer continuity the standalone maps do not
carry. Per-entry proof allows the SDK to use a good composed experience where it
works while retaining the dependable Genes two-layer workflow elsewhere.

Separating debug companions keeps production packages private and conventional
without sacrificing an exact post-deployment investigation artifact. Preserving
native frames and readable PHP also means correlation failure does not make a
WordPress incident opaque.

## Alternatives considered

### Use Source Map v3 as the only PHP format

Rejected as the primary PHP contract. Source Map v3 is mature for generated
line/column positions in browser tooling, but ordinary PHP stacks provide no
column and PHP/WordPress has no universal consumer. It also does not directly
express emitter-owned line anchors, semantic-plan identities, compiler-generated
reasons, or authenticated target bytes. A future optional export is possible,
but the v1 range map remains authoritative.

### Use inline source comments or target line directives only

Rejected as authority. The full port proves comments can help a bounded stock-
Haxe PHP investigation, and the OCaml compiler family proves line directives can
make native compiler errors more actionable. Both are intentionally best-effort
and can alter output layout. Comments/directives do not bind the complete
generated bytes, model nested semantic ranges, or provide a package-wide browser
chain. Safe comments remain a secondary development aid.

### Always compose browser maps

Rejected. Genes explicitly documents that automatic JS-to-TS-to-Haxe composition
is not guaranteed, and build adapters differ in how they load, transform, and
rewrite maps. Composition is admitted only after exact runtime evidence;
two-stage resolution is a supported outcome rather than a failed ambition.

### Resolve PHP stack lines to the nearest mapped expression

Rejected. A line-only frame cannot identify which expression column executed,
and formatting/helper insertion makes nearest-line selection unstable. The
Elixir reference's best-effort nearest-context behavior is useful experimental
precedent but is not adopted as an exact SDK contract. PHP line mapping requires
an explicit trace anchor.

### Make a runtime WordPress handler the primary workflow

Rejected. It would add production behavior, interact with recovery/error-handler
ordering, and make a build artifact dependent on runtime map availability. The
offline CLI is deterministic and non-invasive; a development-only augmenting
handler can be layered on later.

### Keep only readable generated target source

Limited fallback, not the selected complete architecture. RubyHx demonstrates
that honest readable generated output and native backtraces are preferable to
an unsupported remapping claim. WordPressHx keeps the same fallback, but its PRD
also requires source correlation, so content-bound PHP maps and layered browser
maps proceed behind explicit evidence states.

### Defer or omit source correlation

Rejected. It would leave Haxe-first site authors unable to diagnose native
failures at the compiler boundary and would postpone a public lookup contract
until after emitters and packages had already embedded incompatible assumptions.
The stop condition is instead to ship partial/native results honestly.

## Consequences

Positive consequences:

- PHP and browser lookups share one package-level integrity and retention model
  while using appropriate target formats.
- Mappings cannot survive changed target bytes or resolve through basename/path
  guesses.
- PHP line-only correlation is intentionally honest and deterministic.
- Browser composition can improve over time without making two-stage builds
  unsupported.
- Native PHP/browser frames remain available when maps are missing or invalid.
- Debug data is separable from the default production WordPress artifact.

Costs and limitations:

- Exact byte indexing, source hashing, semantic node IDs, and trace anchors add
  compiler and build complexity.
- Fine-grained PHP coverage will initially be incomplete; missing anchors are
  visible rather than backfilled heuristically.
- Debug companions need their own retention, license, secret, and provenance
  checks.
- Map composition must be re-proven per exact adapter/profile/entry/mode after
  relevant toolchain changes.
- The current declaration range types require a deliberate SDK-025 migration.
- This ADR accepts formats and contract fixtures only. PHP runtime mapping, the
  CLI, official WordPress browser correlation, deliberate throws, and production
  support remain untested.

## Migration, rollback, and supersession

SDK-025 must add exact byte provenance to the generic PHP printer/lowering path
without reinterpreting SDK-021's one-based range type. It should first emit and
lookup the neutral fixture, then add deliberate public/private WordPress failures,
package indexing, trace text/JSON snapshots, and privacy negatives.

SDK-034 must project existing SDK-032 evidence into the index, add deliberate
development/minified failures, and independently prove the official
`@wordpress/scripts` lane. An entry whose composition proof regresses rolls back
to `browser-two-stage-v3`; runtime bundle bytes need not change. A PHP map
regression can omit the map and report native/unmapped frames without changing
native exception behavior. A debug-companion privacy problem withdraws that
companion without changing the install artifact.

Readers of v1 remain available for already published debug artifacts. A future
schema that changes coordinates, tie-breaking, trace-anchor meaning, path
identity, or integrity semantics uses a new version and migration. Supersession
must identify affected package/profile receipts and cannot retroactively upgrade
old evidence.
